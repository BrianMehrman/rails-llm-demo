# Implementation Patterns

Concrete examples for common task types. Follow these patterns to stay consistent with the existing codebase.

---

## Add a Model

1. Generate the migration: `bin/rails generate migration CreateThings`
2. Run: `bin/rails db:migrate`
3. Create `app/models/thing.rb`

**Model conventions** — see `app/models/message.rb` as the canonical example:

```ruby
class Thing < ApplicationRecord
  belongs_to :chat                      # foreign key relationships
  validates :name, presence: true       # always validate at the model layer
  validates :status, inclusion: { in: %w[ pending complete ] }

  # Conditional validation: allow blank only in a specific state
  validates :content, presence: true, unless: -> { status == "pending" }
end
```

- Use `ROLES = %w[ ... ].freeze` / `STATUSES = %w[ ... ].freeze` constants for string enums.
- Never edit `db/schema.rb` directly — it's generated by `db:migrate`.

## Add a Controller Action

Controllers are thin — no business logic, no LLM calls. Logic goes in `app/services/`.

```ruby
class ThingsController < ApplicationController
  before_action :set_thing, only: %i[ show edit update destroy ]

  def create
    @thing = Thing.new(thing_params)
    if @thing.save
      redirect_to @thing, notice: "Created."
    else
      render :new, status: :unprocessable_entity
    end
  end

  private

  def set_thing
    @thing = Thing.find(params[:id])
  end

  def thing_params
    params.expect(thing: [ :name, :content ])   # params.expect, not params.require.permit
  end
end
```

- Use `params.expect` (Rails 8 strong params) — not `params.require(:thing).permit(...)`.
- `redirect_to` after mutating actions; `render` with `status: :unprocessable_entity` on failure.
- Add routes to `config/routes.rb` using nested resources where appropriate.

## Add a Background Job

Jobs live in `app/jobs/`. See `app/jobs/llm_response_job.rb` as the canonical example.

```ruby
class MyJob < ApplicationJob
  queue_as :default

  def perform(record_id, dependent_id)
    record = MyModel.find(record_id)
    dependent = OtherModel.find(dependent_id)

    begin
      result = MyService.new.call(record)
      dependent.update!(content: result, status: "complete")
    rescue MyService::Error => e
      dependent.update!(content: e.message, status: "error")
    end

    Turbo::StreamsChannel.broadcast_replace_to(
      record,
      target: dependent,
      partial: "things/thing",
      locals: { thing: dependent }
    )
  end
end
```

- Pass IDs, not ActiveRecord objects — jobs are serialized to the queue database.
- Always rescue service errors and update the record's status — never let a job silently fail.
- Broadcast at the end of `perform`, after all DB writes are complete.
- Enqueue with `MyJob.perform_later(record.id, dependent.id)` — never `perform_now` in controllers.

## Add a Turbo Broadcast

The pattern: create a partial, broadcast to a subscribed channel, let the view handle rendering.

**Partial** (`app/views/things/_thing.html.erb`):
```erb
<%= turbo_frame_tag dom_id(thing) do %>
  <div class="thing thing--<%= thing.status %>">
    <%= thing.content.presence || "Thinking…" %>
  </div>
<% end %>
```

**Broadcast** (from the job):
```ruby
Turbo::StreamsChannel.broadcast_replace_to(
  chat,               # the subscribed object (matches the view's turbo_stream_from)
  target: thing,      # dom_id(thing) — the element to replace
  partial: "things/thing",
  locals: { thing: thing }
)
```

**View subscription** (`app/views/chats/show.html.erb`):
```erb
<%= turbo_stream_from @chat %>
```

Rules:
- Never put inline HTML strings in jobs or controllers — always use a partial.
- The `target:` key accepts an ActiveRecord object; Rails calls `dom_id` on it automatically.
- Use `broadcast_replace_to` to swap an existing DOM element; use `broadcast_append_to` to add a new one.

## Add Observability Instrumentation

### OpenTelemetry span

Wrap a logical operation in a named span. See `LlmClient#chat` for the canonical example.

```ruby
def call(input)
  tracer = OpenTelemetry.tracer_provider.tracer("my_service")
  tracer.in_span(
    "my_service.call",
    attributes: {
      "my_service.input_length" => input.length
    }
  ) do |span|
    result = do_work(input)
    span.set_attribute("my_service.result_length", result.length)
    result
  end
end
```

- OTEL is a no-op unless `OTEL_ENABLED=true` — no need to guard calls with `if ENV["OTEL_ENABLED"]`.
- Name spans as `service_name.operation` (snake_case, dot-separated).
- Set attributes on the span for values you'll want to filter/group by in Jaeger.

### Prometheus metric

Register metrics at the class level with the `begin/rescue AlreadyRegisteredError` pattern. See `LlmClient` for the canonical example.

```ruby
REGISTRY = Prometheus::Client.registry
MY_DURATION = begin
  REGISTRY.histogram(
    :my_operation_duration_seconds,
    docstring: "Description of what this measures",
    labels: [ :status ]
  )
rescue Prometheus::Client::Registry::AlreadyRegisteredError
  REGISTRY.get(:my_operation_duration_seconds)
end
```

Then observe in your method:

```ruby
start = Process.clock_gettime(Process::CLOCK_MONOTONIC)
# ... do work ...
duration = Process.clock_gettime(Process::CLOCK_MONOTONIC) - start
MY_DURATION.observe(duration, labels: { status: "success" })
```

- `/metrics` is always available — Prometheus middleware is mounted unconditionally.
- Use histograms for durations, counters for event counts.
- Always include a `status` label so success/error paths are distinguishable.
